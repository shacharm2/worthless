## Summary

Phase 04.2 transforms the existing test suite into a production-grade CI pipeline with 5 tiers, coverage gates, flaky test quarantine, and GHA workflow files. The project already has a mature test infrastructure (35+ test files, xdist, Hypothesis, respx, syrupy, mutmut) and two basic GHA workflows (tests.yml, sast.yml). The work is primarily wiring and hardening, not greenfield.

The main challenges are: (1) wiring 5 GHA tiers without secrets (zero-token constraint), (2) implementing coverage gating with per-module floors (crypto 95%, proxy 85%, overall 80%), (3) adding pytest-rerunfailures for flaky detection without masking real failures, and (4) absorbing 6 WOR-67 children into new tests.

**Primary recommendation:** Build GHA workflows tier-by-tier (push first, then PR extras, then scheduled/pre-release/manual), with coverage gating via `--cov-fail-under` and `py-cov-action/python-coverage-comment-action` for PR comments.

### Pattern 1: Tiered GHA Workflow (tests.yml rewrite)
**What:** Single workflow file handles Tier 1 (push) and Tier 2 (PR extras) using conditional jobs.
**When to use:** When push and PR tiers share setup but PR adds coverage comment + threshold gate.

```yaml
# Tier 1: Every push
jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-python@v6
        with:
          python-version: "3.13"
      - run: pip install uv
      - run: uv sync --extra dev --extra test
      - run: uv run pytest --reruns 1 --cov=worthless --cov-report=xml --cov-report=term-missing -n auto --dist loadscope
      - run: uv run ruff check .

  # Tier 2: PR-only extras
  coverage-comment:
    if: github.event_name == 'pull_request'
    needs: test
    # ... py-cov-action/python-coverage-comment-action
```

### Pattern 2: Coverage Gate with Per-Module Floors
**What:** Use pytest-cov `--cov-fail-under=80` for overall, then a separate step to parse XML for per-module floors.
**When to use:** When you need both global and per-module thresholds.

```bash
# Overall gate (built into pytest-cov)
uv run pytest --cov=worthless --cov-fail-under=80

# Per-module gate (script parses coverage XML)
python -c "
import xml.etree.ElementTree as ET
tree = ET.parse('coverage.xml')
for pkg in tree.findall('.//package'):
    name = pkg.get('name', '')
    rate = float(pkg.get('line-rate', 0)) * 100
    if 'crypto' in name and rate < 95:
        raise SystemExit(f'crypto coverage {rate:.1f}% < 95%')
    if 'proxy' in name and rate < 85:
        raise SystemExit(f'proxy coverage {rate:.1f}% < 85%')
"
```

### Pattern 4: Flaky Test Quarantine
**What:** pytest-rerunfailures with `--reruns 1` detects flakiness. Quarantined tests get `@pytest.mark.quarantine` and run in separate non-blocking job.
**When to use:** When a test fails once then passes on rerun -- that is a flaky signal.

```python
# Custom marker
@pytest.mark.quarantine
def test_flaky_thing():
    ...
```

```yaml
# GHA: quarantine job (non-blocking, continue-on-error)
quarantine:
  continue-on-error: true
  steps:
    - run: uv run pytest -m quarantine --reruns 1
```

### Pattern 5: xdist loadscope
**What:** Change from default `load` to `loadscope` to group tests by module.
**When to use:** When module-level fixtures (like DB setup) cause isolation issues.

```toml
# pyproject.toml change
addopts = "--strict-config --strict-markers -n auto --dist loadscope"
```

### Anti-Patterns to Avoid
- **Retries hiding real failures:** Never use `--reruns > 1`. One rerun is a detector, not a fix.
- **Coverage as only quality metric:** Coverage gates catch gaps but don't prove correctness. Mutation testing fills that role.
- **Secrets in CI:** All tools must work with zero GitHub secrets. No SEMGREP_APP_TOKEN, no Codecov token.
- **Scheduled jobs running on main only:** Scheduled workflows only run on the default branch. Don't try to schedule against feature branches.

## Common Pitfalls

### Pitfall 1: xdist + coverage interaction
**What goes wrong:** Coverage under xdist can miss lines because each worker only sees its subset.
**Why it happens:** pytest-cov needs `--cov-append` or the `parallel` option to combine worker results.
**How to avoid:** Set `[tool.coverage.run] parallel = true` and add `concurrency = ["multiprocessing"]` in pyproject.toml. pytest-cov handles combine automatically.
**Warning signs:** Coverage numbers drop when running with `-n auto` vs single-process.

### Pitfall 2: pytest-rerunfailures + xdist incompatibility
**What goes wrong:** Some versions have issues with xdist parallel reruns.
**Why it happens:** Worker scheduling and rerun scheduling can conflict.
**How to avoid:** Use pytest-rerunfailures >=14.0 which has xdist compatibility fixes. Test locally with `-n auto --reruns 1`.
**Warning signs:** Tests pass locally but show "rerun" status in CI with no actual flakiness.

### Pitfall 3: Scheduled GHA cron drift
**What goes wrong:** Scheduled workflows may not run exactly on time, and they ONLY run on the default branch.
**Why it happens:** GitHub Actions schedules are approximate and only trigger on default branch.
**How to avoid:** Accept delay (up to 15 min), and ensure scheduled workflow is merged to main before expecting it to run.
**Warning signs:** Scheduled jobs never trigger on feature branches.

### Pitfall 4: py-cov-action needs write permissions
**What goes wrong:** Coverage comment action fails with 403 on PR from fork.
**Why it happens:** Pull requests from forks don't get write permissions by default.
**How to avoid:** Use `workflow_run` trigger pattern for the comment step, or limit to same-repo PRs (fine for private repo).
**Warning signs:** Action fails on external contributor PRs.

### Pitfall 5: semgrep --config auto needs network
**What goes wrong:** semgrep with `--config auto` downloads rules at runtime.
**Why it happens:** `--config auto` fetches community rulesets from semgrep.dev.
**How to avoid:** This is fine in CI (has network). For local pre-commit, bandit runs offline. Semgrep is CI-only by design (already documented in .pre-commit-config.yaml).
**Warning signs:** Pre-commit semgrep failing without network.

### Pitfall 6: Hypothesis database=None + derandomize interaction
**What goes wrong:** Without database, Hypothesis cannot save failing examples for replay.
**Why it happens:** `database=None` disables the example database; `derandomize=True` makes it deterministic anyway.
**How to avoid:** This is intentional in CI. Locally, keep database enabled for development. The ci profile should only activate in GHA.
**Warning signs:** None -- this is the correct pattern for CI.

### Per-module coverage gate script (scripts/check-coverage-floors.py)
```python
#!/usr/bin/env python3
"""Check per-module coverage floors from coverage.xml."""
import sys
import xml.etree.ElementTree as ET

FLOORS = {
    "worthless.crypto": 95.0,
    "worthless.proxy": 85.0,
}

tree = ET.parse("coverage.xml")
failed = False
for pkg in tree.findall(".//package"):
    name = pkg.get("name", "").replace("/", ".")
    rate = float(pkg.get("line-rate", 0)) * 100
    for module, floor in FLOORS.items():
        if module in name and rate < floor:
            print(f"FAIL: {name} coverage {rate:.1f}% < {floor}%")
            failed = True

sys.exit(1 if failed else 0)
```

## Open Questions

1. **Quarantine automation**
   - What we know: Policy says auto-quarantine after 2 flaky runs in 7 days
   - What's unclear: Automation requires tracking flaky history -- GHA doesn't have built-in flaky tracking
   - Recommendation: Start with manual quarantine (add marker + file beads issue). Automate later if volume warrants it.

2. **py-cov-action permissions on private repo**
   - What we know: Works without external tokens, needs `permissions: pull-requests: write`
   - What's unclear: Exact workflow_run pattern needed if repo goes public with fork PRs
   - Recommendation: Use simple permissions for now (private repo), document upgrade path.

3. **semgrep rule set for push tier**
   - What we know: `--config auto` downloads community rules (needs network, ~2800 rules)
   - What's unclear: Whether `--config auto` is fast enough for push tier (<5 min target)
   - Recommendation: Use `--config auto` initially, profile in CI, switch to `--config p/python` subset if too slow.
