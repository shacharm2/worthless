# Phase 03.1: Proxy Hardening - Research

**Researched:** 2026-03-21
**Domain:** FastAPI proxy security hardening, SSE streaming, SQLite atomicity
**Confidence:** HIGH

## Summary

Phase 03.1 fixes all blocker, high, and medium-severity findings from the Phase 3 multi-agent review. The codebase already implements the core proxy pipeline correctly, but has critical gaps: SSE responses are buffered (not streamed), Fernet decryption happens before the rules engine gate (CRYP-05 violation), shard material uses `bytes` instead of `bytearray` (SR-01 violation), and several race conditions exist in spend tracking and file access.

The fixes are well-scoped refactors -- no new features, no new dependencies (except possibly `aiosqlitepool` for connection pooling). Every fix has a clear pattern from the existing codebase or standard library. The highest-risk change is B-1 (SSE streaming) because it requires restructuring the `secure_key` context scope to overlap with the streaming lifecycle.

**Primary recommendation:** Fix in blocker-first order per CONTEXT.md. B-1 and B-2 are the hardest -- tackle them first, then the remaining fixes are mechanical.

## Architecture Patterns

### Pattern 1: SSE Streaming with Secure Key Lifecycle (B-1)

**What:** The `secure_key` context must encompass the entire streaming lifecycle. Currently, httpx sends the request inside `secure_key` but the response is consumed outside it.

**When to use:** Any time key material is needed during a long-lived operation.

**Approach:**
```python
# CURRENT (broken): key zeroed before stream consumed
with secure_key(key_buf) as k:
    upstream_resp = await httpx_client.request(...)  # no stream=True
# key is zeroed here, but stream hasn't been read yet

# FIXED: use client.send() with stream=True, consume inside context
with secure_key(key_buf) as k:
    adapter_req = adapter.prepare_request(body=body, headers=req_headers, api_key=k)
    # Build httpx.Request manually
    req = httpx_client.build_request(
        method=request.method,
        url=adapter_req.url,
        headers=adapter_req.headers,
        content=adapter_req.body,
    )
    upstream_resp = await httpx_client.send(req, stream=True)
# Key zeroed here -- but for streaming, we need the response alive

# The key insight: key material is only needed for the SEND, not for reading
# the stream. So secure_key wraps just the send, then we read stream after.
# httpx stream response stays open via manual mode (no context manager).
```

**Critical insight:** The API key is only embedded in the request headers by `prepare_request()`. Once `httpx_client.send()` dispatches the request, the key is no longer needed. The stream can be read AFTER `secure_key` exits. The fix is:
1. Use `httpx_client.send(req, stream=True)` inside `secure_key`
2. Exit `secure_key` (key zeroed)
3. Return `StreamingResponse` that reads from `upstream_resp.aiter_bytes()`
4. Use `BackgroundTask` to call `upstream_resp.aclose()` after stream completes

```python
with secure_key(key_buf) as k:
    adapter_req = adapter.prepare_request(body=body, headers=req_headers, api_key=k)
    req = httpx_client.build_request(
        method=request.method,
        url=adapter_req.url,
        headers=adapter_req.headers,
        content=adapter_req.body,
    )
    upstream_resp = await httpx_client.send(req, stream=True)
# key_buf zeroed here -- upstream_resp still open for reading

# For streaming:
async def _stream_with_metering() -> AsyncIterator[bytes]:
    try:
        async for chunk in upstream_resp.aiter_bytes():
            collected_chunks.append(chunk)
            yield chunk
    finally:
        await upstream_resp.aclose()  # cleanup on disconnect or completion

# For non-streaming: read body, then close
body = await upstream_resp.aread()
await upstream_resp.aclose()
```

### Pattern 2: Split Repository for Gate-Before-Decrypt (B-2)

**What:** Split `ShardRepository.retrieve()` into two methods to enforce gate ordering.

```python
class EncryptedShard(NamedTuple):
    """Raw encrypted shard -- NOT decrypted yet."""
    shard_b_enc: bytes
    commitment: bytes
    nonce: bytes
    provider: str

class ShardRepository:
    async def fetch_encrypted(self, alias: str) -> EncryptedShard | None:
        """Fetch encrypted shard record. No Fernet decrypt."""
        async with aiosqlite.connect(self._db_path) as db:
            # ... SELECT shard_b_enc, commitment, nonce, provider
            return EncryptedShard(row["shard_b_enc"], ...)

    def decrypt_shard(self, encrypted: EncryptedShard) -> StoredShard:
        """Fernet-decrypt shard_b. Call ONLY after gate passes."""
        shard_b = bytearray(self._fernet.decrypt(encrypted.shard_b_enc))
        return StoredShard(
            shard_b=shard_b,
            commitment=bytearray(encrypted.commitment),
            nonce=bytearray(encrypted.nonce),
            provider=encrypted.provider,
        )
```

**App.py flow becomes:**
```python
# (f) Fetch encrypted (no decrypt)
encrypted = await repo.fetch_encrypted(alias)
if encrypted is None: return 401

# (h) GATE: rules engine evaluates BEFORE any decrypt
denial = await rules_engine.evaluate(alias, request)
if denial is not None: return denial

# (i) NOW decrypt (gate passed)
stored = repo.decrypt_shard(encrypted)
```

### Pattern 3: Atomic Spend Check-and-Increment (H-3)

**What:** Use a single SQL statement with a subquery to atomically check and record spend.

```python
async def evaluate(self, alias: str, request: object) -> ErrorResponse | None:
    async with aiosqlite.connect(self.db_path) as db:
        # Single atomic query: check cap AND get current spend
        row = await (await db.execute(
            """
            SELECT ec.spend_cap, COALESCE(
                (SELECT SUM(tokens) FROM spend_log WHERE key_alias = ?), 0
            ) as total_tokens
            FROM enrollment_config ec
            WHERE ec.key_alias = ?
            """,
            (alias, alias),
        )).fetchone()

        if row is None:
            return None  # No config = no cap
        spend_cap, total_tokens = row
        if spend_cap is None:
            return None  # NULL cap = unlimited
        if total_tokens >= spend_cap:
            return spend_cap_error_response()
    return None
```

**Note:** The real TOCTOU fix requires the check and the subsequent INSERT into spend_log to happen in the same transaction. With SQLite WAL mode + IMMEDIATE transaction, the write lock prevents concurrent modifications between the check and the insert. The spend check in the rules engine and the spend recording in metering must share a transaction or use `BEGIN IMMEDIATE`.

**Recommended simpler approach:** Since SQLite serializes writes anyway with WAL mode, the actual risk window is small. For the PoC, wrapping the spend check in `BEGIN IMMEDIATE` is sufficient to prevent concurrent requests from both passing the check.

### Pattern 4: Constant-Time Alias Lookup (H-2)

**What:** Ensure invalid alias takes the same code path and timing as valid alias.

```python
# Current problem: if alias not found, we return 401 immediately
# Valid alias continues to load shard_a, evaluate rules, etc.
# Fix: always perform the same operations regardless of alias validity

# The DB query itself is already constant-time (index lookup)
# The timing leak is in the subsequent operations

# Option A: Always fetch, always evaluate rules, then check validity
encrypted = await repo.fetch_encrypted(alias)
# ... load shard_a ...
# ... evaluate rules ...
# THEN check if encrypted is None

# Option B (simpler): Use hmac.compare_digest on the alias lookup
# to prevent timing differences in the string comparison itself
# But the real issue is the code path divergence, not string comparison

# Recommended: Restructure so the 401 return happens at the same point
# regardless of which check failed. Already partially done with uniform 401.
# Main fix: ensure same DB query pattern runs for invalid aliases.
```

**Practical approach:** The anti-enumeration requirement is about response format uniformity (already done with `_AUTH_BODY`) and timing uniformity. For timing: ensure the DB lookup always happens (it already does -- `repo.fetch_encrypted()` runs for every request with an alias). The main leak is that invalid alias skips shard_a loading and rules evaluation. Fix by moving the None check to after all validation steps complete.

### Pattern 5: Request Body Size Limit (M-1)

**What:** Limit request body to prevent OOM.

```python
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.responses import Response

class BodySizeLimitMiddleware(BaseHTTPMiddleware):
    def __init__(self, app, max_bytes: int = 10 * 1024 * 1024):  # 10 MB default
        super().__init__(app)
        self.max_bytes = max_bytes

    async def dispatch(self, request: Request, call_next):
        content_length = request.headers.get("content-length")
        if content_length and int(content_length) > self.max_bytes:
            return Response(
                content=b'{"error": {"message": "request too large", "type": "invalid_request_error"}}',
                status_code=413,
                media_type="application/json",
            )
        return await call_next(request)
```

**Recommended default:** 10 MB. LLM API request bodies are typically < 1 MB even with long prompts. 10 MB provides generous headroom.

### Anti-Patterns to Avoid
- **Streaming inside secure_key:** Don't try to keep `secure_key` open during the entire stream -- the key is only needed for the send, not the read
- **Manual connection management without cleanup:** Always use `try/finally` with `upstream_resp.aclose()` for streamed responses
- **aiosqlite connection per operation:** Current pattern opens 3 connections per request; switch to persistent connection
- **check-then-act without transaction:** Always wrap check+act in a single `BEGIN IMMEDIATE` transaction for atomicity

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CORS denial | Custom header stripping | `CORSMiddleware(allow_origins=[])` | Handles preflight, varies-origin, edge cases |
| Constant-time compare | Custom byte loop | `hmac.compare_digest()` | C-level implementation, audited, handles length leaks |
| Rate limiter cleanup | Manual timer thread | `asyncio.create_task` with periodic cleanup | Integrates with event loop, no threading complexity |
| Body size limiting | Raw ASGI middleware | Starlette `BaseHTTPMiddleware` with content-length check | Handles edge cases, integrates with FastAPI |

## Common Pitfalls

### Pitfall 1: Streaming Response Cleanup
**What goes wrong:** httpx response not closed when client disconnects mid-stream
**Why it happens:** `StreamingResponse` generator gets cancelled but no `finally` block closes the upstream response
**How to avoid:** Always use `try/finally` in the stream generator to call `await upstream_resp.aclose()`
**Warning signs:** Connection pool exhaustion under load, "too many open connections" errors

### Pitfall 2: secure_key Scope vs Stream Lifetime
**What goes wrong:** Trying to keep `secure_key` open for the duration of streaming (could be minutes)
**Why it happens:** Misunderstanding that the key is needed for reading, not just sending
**How to avoid:** Key is only in the request headers. Once httpx sends the request, the key buffer can be zeroed. Stream reading doesn't need the key.
**Warning signs:** Key material held in memory for entire stream duration

### Pitfall 3: SQLite IMMEDIATE vs DEFERRED
**What goes wrong:** Two concurrent requests both pass the spend check, both record spend, budget exceeded
**Why it happens:** Default `DEFERRED` transactions don't acquire the write lock until the INSERT
**How to avoid:** Use `BEGIN IMMEDIATE` for spend-check transactions so the write lock is acquired at transaction start
**Warning signs:** Spend slightly exceeding cap under concurrent load

### Pitfall 4: bytearray in NamedTuple
**What goes wrong:** `StoredShard` is a `NamedTuple` with `bytes` fields -- can't hold `bytearray`
**Why it happens:** NamedTuple type annotations are documentation only, not enforced
**How to avoid:** Change type annotations to `bytearray`, ensure all construction sites pass `bytearray`. Consider switching to `dataclass` if zeroing logic is needed on the type itself.
**Warning signs:** `isinstance` checks failing, zeroing attempts on `bytes` objects

### Pitfall 5: Rate Limiter Memory Growth
**What goes wrong:** `defaultdict(list)` grows without bound as unique (alias, ip) pairs accumulate
**Why it happens:** No cleanup of expired entries
**How to avoid:** Periodic cleanup task that removes entries older than the window. Run every 60 seconds via `asyncio.create_task` in lifespan.
**Warning signs:** Memory growth over days of operation

### Pitfall 6: Non-Streaming Response with stream=True
**What goes wrong:** When upstream returns a non-streaming response but we used `stream=True`, must explicitly read and close
**Why it happens:** `stream=True` defers body reading -- even for JSON responses
**How to avoid:** After detecting non-streaming content-type, call `await upstream_resp.aread()` then `await upstream_resp.aclose()`
**Warning signs:** Response body is empty, connection leaks

### Rate Limiter with TTL Cleanup

```python
@dataclass
class RateLimitRule:
    default_rps: float = 100.0
    _windows: dict[tuple[str, str], list[float]] = field(
        default_factory=dict, init=False, repr=False
    )
    _last_cleanup: float = field(default_factory=time.monotonic, init=False, repr=False)
    _cleanup_interval: float = 60.0  # seconds

    async def evaluate(self, alias: str, request: object) -> ErrorResponse | None:
        now = time.monotonic()

        # Periodic global cleanup
        if now - self._last_cleanup > self._cleanup_interval:
            self._cleanup(now)

        key = (alias, getattr(getattr(request, "client", None), "host", "unknown"))
        window = self._windows.get(key, [])
        cutoff = now - 1.0
        window = [t for t in window if t > cutoff]

        if len(window) >= self.default_rps:
            retry_after = max(1, int(window[0] + 1.0 - now) + 1)
            return rate_limit_error_response(retry_after=retry_after)

        window.append(now)
        self._windows[key] = window
        return None

    def _cleanup(self, now: float) -> None:
        cutoff = now - 2.0  # Keep entries for 2 seconds (generous)
        to_remove = [k for k, v in self._windows.items() if not v or v[-1] < cutoff]
        for k in to_remove:
            del self._windows[k]
        self._last_cleanup = now
```

### Upstream Error Sanitization

def _sanitize_upstream_error(status_code: int, body: bytes, provider: str) -> bytes:
    """Sanitize provider error response -- strip internal details."""
    try:
        parsed = json.loads(body)
        if provider == "anthropic" and isinstance(parsed, dict):
            error = parsed.get("error", {})
            return json.dumps({
                "type": "error",
                "error": {
                    "type": error.get("type", "api_error"),
                    "message": "upstream provider error",
                }
            }).encode()
        elif isinstance(parsed, dict) and "error" in parsed:
            return json.dumps({
                "error": {
                    "message": "upstream provider error",
                    "type": parsed["error"].get("type", "api_error"),
                    "param": None,
                    "code": None,
                }

        pass

    return json.dumps({"error": {"message": "upstream error", "type": "api_error"}}).encode()

## Open Questions

1. **Connection pooling vs persistent connection**
   - What we know: aiosqlitepool exists but adds a dependency; single persistent connection works for PoC
   - What's unclear: Whether a single connection handles concurrent async reads well enough under load
   - Recommendation: Start with single persistent connection. SQLite WAL mode allows concurrent reads. If bottleneck appears, upgrade to pool.

2. **Non-streaming path with stream=True**
   - What we know: Using `stream=True` always means we must explicitly read and close the response
   - What's unclear: Whether we should always use `stream=True` or detect streaming need before sending
   - Recommendation: Always use `stream=True` with `client.send()`. After getting the response, check content-type to decide streaming vs buffered path. This avoids buffering large responses.

3. **Anti-enumeration timing depth**
   - What we know: Response format is already uniform. DB query timing is roughly constant.
   - What's unclear: How much timing equalization is worth pursuing at PoC stage
   - Recommendation: Ensure same code path (DB lookup + shard_a load attempt) runs for all requests. Don't add artificial delays -- they're fragile and can introduce new side channels.
