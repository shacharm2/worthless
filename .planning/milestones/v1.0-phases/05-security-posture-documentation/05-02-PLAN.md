**Document structure (from CONTEXT.md locked decisions):**

1. **Last verified: [date]** + commit SHA at top
2. **Disclaimer**: "This document is not a compliance certification. Worthless is not SOC 2 audited. OWASP references show which vulnerability classes the architecture addresses. Full framework mapping ships with enterprise tier."
3. **TL;DR** — 5 lines max, developer stops here. Brief origin hook mapping to threat model (stolen key -> bill runup, 1-2 sentences). "Stolen? So what." confidence tone.
4. **Table of Contents** with anchor links
5. **Glossary** — define: Shard A, Shard B, commitment, nonce, gate, enrollment, reconstruction. Outside auditors don't share our vocabulary.
6. **Confidence Tier Definitions** — 3-tier table:
   - **Enforced**: Property verified by automated CI tests. Bypass requires modifying test suite (PR review catches). Evidence: test name.
   - **Best-effort**: Implementation correct, but runtime/language constraints prevent full guarantee. Evidence: code path + known gap.
   - **Planned**: Not yet implemented, on Rust hardening roadmap. Evidence: ROADMAP.md reference.
   - Hard cap: no more than 3 items at Best-effort tier. More = architecture problem.
   - Each item includes confidence upgrade path (what changes it from Best-effort to Enforced).
7. **Trust Boundary Diagram** — ASCII art showing:
   - Client boundary (CLI, Shard A, split_key)
   - Network boundary (what crosses: Shard B + commitment + nonce at enrollment, x-worthless-key header at request time)
   - Proxy boundary (rules engine, gate)
   - Reconstruction boundary (decrypt, XOR, upstream call)
   - Label what data crosses each boundary. "Reconstructed key" arrow NEVER crosses reconstruction->proxy.
   - Annotate where each invariant violation would be visually obvious.
8. **Confidence Summary Table** — 3 invariants + key SRs in one table with tier, evidence, and upgrade path. Auditor gets full picture in 1 page.
9. **Per-Invariant Cards** (the core — each is self-contained):
   - **Invariant 1: Client-Side Splitting** — Claim, evidence (cite exact test names from test_invariants.py), confidence tier (Enforced), attacker prerequisites ("must have code execution in the CLI process to intercept split_key input"), limitations (dynamic imports not caught, test-time not compile-time), Rust mitigation (split_key won't exist in reconstruction binary). Reference enforcing SRs inline (SR-01, SR-08).
   - **Invariant 2: Gate Before Reconstruction** — Claim, evidence (cite exact test names from test_security_properties.py), confidence tier (Enforced), attacker prerequisites ("must bypass rules engine evaluation or modify proxy source code"), limitations (in-process means no hardware isolation), Rust mitigation (process boundary + seccomp). Reference enforcing SRs inline (SR-03).
   - **Invariant 3: Server-Side Direct Upstream Call** — Claim, evidence (cite exact test names from test_invariants.py, including new Plan 01 test), confidence tier (Enforced), attacker prerequisites ("must have code execution in the FastAPI process to intercept the reconstructed key before zeroing"), limitations (shared memory space, GC may retain copies), Rust mitigation (distroless container, mlock, zeroize). Reference enforcing SRs inline (SR-01, SR-02, SR-04).
10. **Non-Goals** — explicit about what Worthless does NOT protect against:
    - Compromised client machine (attacker has full access to CLI process)
    - Malicious LLM provider (provider could log/store requests regardless)
    - Side-channel timing attacks on the Python PoC
    - Memory forensics on the proxy host (GC limitation)
    - Supply chain attacks on Python dependencies
    - Compromised proxy server (attacker with shell access can read process memory)
    - Nation-state adversaries with physical access
11. **SR Reverse-Mapping Table** — SR-01 through SR-08, each row: which invariant it enforces, which tests verify it (exact names), where in code (module path, not line numbers). This is the auditor's primary navigation tool.
12. **Known Limitations (Python PoC)** — Per-limitation threat cards:
    - **Memory Safety**: GC non-determinism, no mlock, intermediate bytes objects, no compiler barrier. Each with: what it means, realistic exploit narrative, attacker prerequisites (precise: "code execution in FastAPI process" not "root on proxy host"), risk level, specific Rust mechanism (zeroize crate, mlock, stack-allocated buffers, explicit_bzero).
    - **Process Isolation**: in-process reconstruction, shared memory space. Each with same structure. Rust: separate distroless container, seccomp.
    - **Key Lifecycle**: no bulk rotation in V1, re-enrollment is manual. Breach response: re-enroll all affected keys.
    - **Shard B Data-at-Rest**: document honestly based on actual code inspection. If Fernet-encrypted, say so with evidence. If not, say so as limitation.
    - **Cryptographic Agility**: check if protocol is versioned (shard schema version field). Document status honestly. If not versioned, note it as a gap (file beads issue per CONTEXT.md).
13. **Breach Scenario** — "What if Shard B DB is compromised?" Document: re-enroll all affected keys, no bulk rotation in V1.
14. **Forensic Logging** — document what IS currently logged (check actual code), not aspirational claims. Add recommendations for gaps if found.
15. **Supply Chain** — one line: "Dependency auditing: pip-audit runs in CI. Full SBOM and supply chain policy ships with enterprise tier."
16. **License** — brief implications paragraph (2-3 sentences) for enterprise evaluators.
17. **Residual Risk Summary Table** — accepted risks ranked by severity (High/Medium/Low), with reference to the limitation card that details each.
18. **Update Cadence and Ownership** — phase triggers, ownership model, how to flag sections for review.
19. **Changelog** — initial entry with today's date + commit SHA.

**Style rules (from CONTEXT.md locked decisions):**

- Every claim cites evidence (test file::function, code module, or SR reference)
- No FIPS/SOC2 compliance language
- OWASP references inline where they naturally fit, with the explicit disclaimer
- No dates for Rust timeline — reference ROADMAP.md
