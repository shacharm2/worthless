# Stress Test: v1.1 Before v2.0 Harden

**Date:** 2026-04-06
**Decision under review:** Ship a v1.1 release (9 unfinished v1.0 items + "light mode" branding) before starting v2.0 Harden (Shamir + Rust sidecar).

---

## STEEL MAN

The strongest version of this proposal: Worthless v1.0 shipped 3 days ago with a working split-key proxy but 9 mode-agnostic features missing from the V1 scope (MCP server enrichment, daemon, keys command, 3 rules engine extensions, Redis metering, alerts, scan hook, SKILL.md). These features are needed by *both* light and secure mode users. Building them now means they get tested against the simpler XOR+Fernet architecture first. Building them after sidecar means either (a) building them twice with sidecar compatibility from the start, or (b) retrofitting sidecar compatibility later — both more expensive than building on a stable, well-understood base. Additionally, branding the current architecture as "light mode" before any user encounters it establishes the mental model that `--secure` is additive, not a replacement. A v1.1 release forces a quality gate: you can't ship with known gaps.

This is what a disciplined product team would do: finish what you started, establish the baseline, then extend.

---

## CLAIMS & ASSUMPTIONS

1. **The 9 items are mode-agnostic** — they serve both light and secure users equally. *Falsifiable: check if daemon, keys, MCP change interface when sidecar exists.*
2. **Building after sidecar means building twice** — sidecar fundamentally changes how these features work. *Falsifiable: trace Phase 11 requirements against each item.*
3. **Light mode branding needs to be established early** — users who adopt before v2.0 will expect additive behavior. *Falsifiable: count current users.*
4. **Shipping forces quality** — a release gate catches gaps that "just merge to main" doesn't. *Falsifiable: is there a CI/release pipeline already?*
5. **The 9 items are "finishing v1"** not scope creep. *Falsifiable: were they in the original V1 scope?*
6. **Python holding keys in GC-managed memory is a meaningful risk right now.** *Falsifiable: who is the current user base?*
7. **v1.1 delays Harden by a meaningful amount.** *Falsifiable: estimate effort.*
8. **Redis metering and alerts belong in a local-only tool.** *Hidden assumption: the current tool IS local-only.*

---

## PRE-MORTEM: 6 Months From Now, This Failed

### Failure Mode 1: v1.1 Became v1.5

The 9 items ballooned. Redis metering pulled in Redis as a dependency. Email+Slack alerts needed an event bus. The daemon needed process supervision. Each feature spawned 2-3 subfeatures. By October 2026, v1.1 still hasn't shipped and Harden hasn't started. The window to differentiate Worthless (split-key with proper memory protection) closed. A competitor shipped a Rust-native key proxy.

**Warning sign ignored:** No one estimated the 9 items. "Finish what we started" felt small but was never sized.

**Cost:** 4+ months of sidecar delay. The core security story — "keys never touch GC-managed memory" — remains undeliverable.

### Failure Mode 2: Daemon/Keys/MCP Got Rebuilt Anyway

v1.1 shipped the daemon as a simple `subprocess.Popen` wrapper. v2.0 Phase 11 (PY-01 through PY-15) required the daemon to manage a sidecar child process, health-check it, restart it on crash, and route requests through IPC. The MCP server needed a new `--secure` code path. The `keys` command needed to distinguish Fernet keys from Shamir shards. Every one of these features was rewritten in Phase 11. The "build once" argument was wrong — it was build-once-then-rebuild.

**Warning sign ignored:** Phase 11 has 15 requirements (PY-01 through PY-15) and explicitly says "proxy, CLI, and adapter layer rewired for dual-mode operation." "Rewired" means the interfaces change.

**Cost:** 2-4 weeks of rework. Developer morale hit from throwing away recent work.

### Failure Mode 3: Zero Users, Zero Feedback

v1.1 shipped to nobody. No OpenClaw listing yet. No public announcement. The "shipping forces quality" argument was theater — there was no external forcing function. The release sat on PyPI collecting dust while internal effort went into features nobody validated.

**Warning sign ignored:** "Target audience progression: solo dev dogfood" — the first user is the developer themselves.

**Cost:** Opportunity cost only, but still real. 3-4 weeks spent on features with no user signal.

## RED TEAM

### Attack 1: "The 9 items are mode-agnostic"

**PARTIALLY FALSE.** I traced Phase 11's requirements against the codebase:

- Phase 11, Success Criterion 2: "`worthless up --secure` starts the proxy **and sidecar**, routing requests through IPC for reconstruction" — the **daemon** must manage the sidecar lifecycle. A v1.1 daemon that doesn't know about sidecar child processes will be gutted in Phase 11.
- Phase 11, Success Criterion 4: "`worthless get <alias>` retrieves a reconstructed key via **sidecar vault mode**" — the **keys command** interface changes fundamentally. v1.1 keys would return Fernet-decrypted bytes; v2.0 keys must talk to the sidecar over IPC.
- Phase 11, Success Criterion 5: "`worthless wrap`, provider adapters, and **MCP server** all work identically in both modes" — the MCP server needs dual-mode routing. Building it light-only first means retrofitting the abstraction layer.

**Truly mode-agnostic items:** model_allowlist, token_budget, time_window (rules engine extensions), `worthless scan` pre-commit hook, SKILL.md, email+Slack alerts. These 6 don't touch the reconstruction path.

**NOT mode-agnostic:** daemon, keys command, MCP server enrichment. These 3 touch the reconstruction path and WILL be rewritten in Phase 11.

**Best counter:** You could design daemon/keys/MCP with a backend abstraction from day one (e.g., `ReconstructionBackend` protocol) so the sidecar just slots in as a new backend. **Why it fails:** You'd be designing the sidecar abstraction without the sidecar existing. The interface won't be right. You're guessing at IPC semantics, health check protocols, and failure modes you haven't encountered yet. Premature abstraction is worse than no abstraction — it creates the wrong seam.

### Attack 2: "Building after sidecar means building twice"

**STRAWMAN for the mode-agnostic items. VALID for the reconstruction-path items — but inverted.**

The rules engine extensions (model_allowlist, token_budget, time_window) evaluate BEFORE reconstruction (that's the whole point of gate-before-reconstruct, SR-03). They don't care whether reconstruction uses Fernet or Shamir. The existing `Rule` protocol in `rules.py:23` is clean — new rules implement `evaluate()` and slot into the `RulesEngine` chain. Zero sidecar dependency. Build once, done.

The scan hook and SKILL.md are pure CLI/docs. Zero reconstruction dependency.

But for daemon/keys/MCP, the "build twice" argument is **backwards**: building them now means building twice, because Phase 11 rewires them. Building them after Phase 11 means building once with the right abstractions.

**Best counter:** The v1.1 daemon could be intentionally minimal (just process management) and Phase 11 extends it. **Why it partially holds:** If the daemon is truly just `start/stop/health-check a child process`, that's reusable. But the MCP server and keys command have interface contracts that change.

### Attack 3: "Light mode branding needs to be established early"

**STRAWMAN.** There are zero external users. The branding argument only matters when users exist who could form expectations. You are the only user. You can brand it "light mode" on the day v2.0 ships and no one will notice the rename.

**Best counter:** If you plan to list on OpenClaw or announce publicly before v2.0, then early branding matters. **Test:** Is there a public launch planned before v2.0?

### Attack 4: "Shipping forces quality"

**PARTIALLY TRUE but misapplied.** The project already has pre-commit hooks (gitleaks, semgrep, trufflehog), CI coverage gates, and a formal verification pipeline (VALIDATION.md, VERIFICATION.md, UAT). The quality forcing function already exists. A PyPI release adds `python -m build` and a version tag — not a meaningfully higher quality bar.

**Best counter:** A release forces you to write real installation docs and test the cold-start experience. That's genuinely valuable. **Why it's not decisive:** You can test the cold-start experience without cutting a release.

### Attack 5: "Every week without sidecar is a week Python holds keys in GC-managed memory"

**STRAWMAN right now.** The only user is the developer. The threat model for "API key briefly exists in Python heap" requires an attacker with memory read access to the process — at which point they can also read the Fernet key, the SQLite DB, and everything else. The sidecar's mlock+MADV_DONTDUMP protection matters for production multi-tenant deployments, not for solo dev dogfood.

**Best counter:** If Worthless gets early adopters before v2.0, this becomes real. **Test:** Is there a public launch planned before Harden completes?

### Attack 6: "Redis metering and alerts don't belong in a local-only tool"

**VALID.** Redis is a network service. The current architecture is SQLite-only, single-process. Adding Redis as a dependency for metering changes the deployment model fundamentally. Email+Slack alerts require outbound credentials and a notification service. These are SaaS-tier features that the CLAUDE.md V1 scope lists but that actively conflict with the "solo dev, 90-second setup" target.

**Best counter:** Redis could be optional (SQLite default, Redis when configured). **Why it weakens the case:** Optional Redis is scope creep with a maintenance surface. You're building a feature for a user segment (teams) that isn't the current target (solo dev dogfood).

### WEAKEST LINK

**The assumption that daemon/keys/MCP are mode-agnostic.** They aren't. Phase 11 has 15 requirements that explicitly rewire these features. Building them light-only now creates interfaces that will be torn out in 2-3 months. This is the single fact that breaks the "build once" argument for 3 of the 9 items.

---

## CONDITIONS FOR VALIDITY

- **v1.1 works IF:** you scope it to the 6 truly mode-agnostic items (rules engine extensions, scan hook, SKILL.md, alerts) and defer daemon/keys/MCP to Phase 11.
- **v1.1 works IF:** you hard-cap it at 2 weeks calendar time with a kill switch: if it's not shippable in 2 weeks, stop and start Harden.
- **v1.1 works IF:** Redis metering is cut (keep SQLite metering, which already exists).
- **Full v1.1 (all 9 items) works IF:** you design daemon/keys/MCP with a `ReconstructionBackend` abstraction AND you have high confidence in the sidecar's IPC interface before writing code. This requires completing Phase 8 research first, which defeats the purpose of shipping v1.1 first.
- **v1.1 breaks WHEN:** any item takes longer than estimated, because each week of v1.1 is a week not spent on the differentiating feature (memory-safe reconstruction).

---

## VERDICT: MODIFY

The core insight is sound — finish mode-agnostic work before adding sidecar complexity. But the proposal as stated includes 3 items (daemon, keys, MCP) that are NOT mode-agnostic and WILL be rewritten in Phase 11, plus 2 items (Redis metering, alerts) that are premature for the current user base.

**Ship a v1.1, but scope it to 5 items:**

1. **Rules engine: model_allowlist** — slots into existing `Rule` protocol, zero sidecar dependency
2. **Rules engine: token_budget** — same
3. **Rules engine: time_window** — same
4. **`worthless scan` pre-commit hook** — pure CLI, exists as scanner.py already
5. **SKILL.md agent discovery** — pure docs

**Defer to v2.0 Phase 11:**
- daemon, keys command, MCP server enrichment (reconstruction-path dependent)

**Cut entirely from v1.x:**
- Redis hot-path metering (premature; SQLite metering already works)
- Email + Slack alerts (SaaS-tier feature; no users to alert)

**Time-box:** 1 week. These 5 items are small — the rules engine extensions are new `Rule` implementations against an existing protocol. If they're not done in a week, something is wrong with the estimate.

**Brand "light mode":** Do it, but as a commit in the v2.0 branch, not as a separate release. The branding only matters when `--secure` exists as a contrast.

**Monitor for:** Scope creep on rules engine items. Each rule should be <100 LOC against the existing protocol. If any item requires schema changes or new dependencies, re-evaluate.
