# Security Panel — Incident-Responder findings (11 gaps)

> **Crash-survival cache.** Promoted to SECURITY-PANEL.md after synthesis.

**Total: 11 gaps. 3 CRIT / 4 HIGH / 3 MED / 1 LOW.**

**Single most damaging "not logged that should be":** the **resolved gpg-agent socket path + binary SHA256 actually used at each sign call** is not in audit line. R-7/R-20 verify them at preflight, but audit log records "tag signed" without the witnessed values — investigator cannot prove WHICH gpg-agent produced the signature.

**Verdict:** spec is **strong on prevention** (R-1/R-3/R-19/R-21/R-24 excellent) but **weak on post-breach reconstruction**. Assumes laptop is intact at incident time.

## §1 Forensic completeness — CRIT

R-15 logs events but line format unspecified. Missing fields that matter post-breach:
- Resolved `$(command -v gh|gpg|docker|pip)` + SHA256 at EACH call (P11 one-shot; binary swappable between P11 and step 4.2b)
- `gpgconf --list-dirs agent-socket` value witnessed at each sign (R-7 checks once)
- `GH_TOKEN` scope at moment of call
- Full `gh api` response bodies for ruleset GET/PATCH/PUT
- Watchdog PID + heartbeat PID + `kill -0` poll timestamps
- `umask`, real UID, login session ID, `who am i` tty
- `git rev-parse HEAD` of the release.sh repo at exec time
- Environment variable snapshot keys filtered by R-13 redactor

**New R-26 (Witnessed Audit Schema):** Define `.release-audit/YYYY-MM-DD.log` as JSONL with mandatory fields `{ts_utc, event, actor_uid, tty, ppid, ppid_comm, tool, tool_path, tool_sha256, gpg_agent_socket, gh_scopes, ruleset_etag, response_sha256, watchdog_pid, expected_sha, actual_sha, redacted_env_keys[]}`. `release-doctor.sh --replay-day YYYY-MM-DD` parses and surfaces gaps.

## §2 Evidence chain of custody — CRIT

`chflags uappnd` + daily `.asc` only protect against LOCAL rewrite. The laptop IS the attacker's playground in a compromise: `chflags nouappnd` as user, rewrite, re-sign with stolen subkey (gpg-agent cached!), pristine. **Zero offsite copy.** Signed tag lives on GitHub; audit log doesn't.

**New R-27 (Offsite Audit Shipping):** After each `release.sh` exit, `gh api -X POST .../issues/<AUDIT_TRACKER>/comments` posting `.asc` (small, signed, public-safe) AND `gh gist create --private` JSONL log; record both URLs locally. Optional: `git notes --ref=audit add` on signed tag, pushed. Three independent copies; any one recoverable months later.

## §3 Reproducibility of forensic artifacts — HIGH

**Strong leg:** investigator months later pulls PyPI wheel + Sigstore bundle, `gh attestation verify` chains to GHA OIDC run + tag ref.

**Weak legs:**
- Tag message is only on-chain narrative; R-16 sanitizes but doesn't require structured fields
- GH Release body reads from local `notes.md` — investigator can't prove it matched repo CHANGELOG
- No PyPI-side artifact lists expected wheel SHA256 for cross-check

**New R-28 (Tag Provenance Trailer):** R-16 extended; tag message MUST end with `--- Provenance ---` trailer containing `Release-SHA: <git>`, `Wheel-SHA256: <hex>`, `Builder-Run-URL: <gh-run>`, `Audit-Manifest: <gist-url>`. Investigator pulls tag from any clone, reads trailer, verifies all four independently.

## §4 Recovery readiness mid-window — CRIT

`release-recover.sh` opens 120s ruleset-disabled window. Laptop compromised at second 30 → watchdog dies with parent (`kill -TERM $$` from heartbeat won't fire on `kill -9` of whole process tree) → ruleset stays disabled for whoever notices first. IR arriving cold has: no way to know which tag was mid-push; no way to know if ruleset re-enabled (needs laptop); no external alarm.

**New R-29 (Pre-Window Beacon):** Before R2 PATCH, script MUST (a) post GitHub issue comment `RULESET-DISABLED v<version> START <iso> PID <pid> EXPECT-RESTORE-BY <iso+120s>` to designated audit issue; (b) trigger GHA scheduled workflow `audit-ruleset-watchdog.yml` with deadline — that workflow runs `gh api /rulesets/15719679` and re-enables from snapshot if deadline passes and no `RULESET-RESTORED` comment exists. **Offsite failsafe, independent of laptop.** IR reads issue thread → exact tag + exact window.

## §5 Communication readiness — HIGH

Linear-paste markdown (4.9) is PROSE. IR has to read it, not grep. No structured handoff for downstream automation (SIEM, ticketing, ChatOps).

**New R-30 (Structured IR Sidecar):** Alongside human markdown, 4.9 MUST also emit `.release-state/<v>/ir-sidecar.json` with `{schema_version, release_version, tag_sha, wheel_sha256, attestation_bundle_url, gh_run_url, audit_log_uri, audit_log_asc_uri, ruleset_window_start, ruleset_window_end, recovery_invoked: bool, doctor_final_state}`. Append one line to repo-tracked `.release-audit/INDEX.jsonl` — `jq` queries across all releases work.

---

## Bottom line

Prevention: **A-**. Forensics: **C**. The five amendments R-26..R-30 close the gap from "we can replay this if the laptop survives" to "any investigator with GitHub read-access can reconstruct timeline cold." **R-29 is non-optional** — a 120s window with no offsite failsafe is single-point-of-failure mirroring prevention-side R-3.
