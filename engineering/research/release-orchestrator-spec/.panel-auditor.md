## §4 SLSA assessment

**Currently meets SLSA Build L3:** signed provenance (Sigstore via Trusted Publisher), hosted build platform (GHA OIDC), isolated build, R-10 verifies bundle chains to repo + tag ref.

**To reach Build L4:**
- (a) two-party review of every change reaching the build — R-14 self-pin is reviewer-gated for scripts but ordinary `main` commits don't require two reviewers per CODEOWNERS
- (b) hermetic, reproducible builds — wheel build not asserted reproducible
- (c) parameterless builds — `publish.yml` inputs not asserted constant

**Source track:** SLSA Source L2 (signed history via `v-tags-signed`) — L3 needs enforced two-person review + retained source.

## §5 Audit-readiness gaps (prioritized)

1. **n=1 signer / no segregation of duties (CC8.1, ISO A.6.1.2).** Add R-26: release requires co-signer attestation (second maintainer GPG-signs `release-approval-<v>.txt` in `.release-audit/`, doctor verifies both fingerprints). Without this, every SOC 2 sample fails.
2. **No SBOM or CVE scan gate before tag-push (PCI 6.3.2, SSDF RV.1).** Add P12 preflight: `pip-audit` + `syft` SBOM generation, fail on HIGH/CRITICAL CVE, attach SBOM to GH Release in step 4.6.
3. **Audit log local-only (CC7.2 monitoring).** Add step 4.9b: ship signed log to immutable WORM store (S3 Object Lock or private audit-repo GH Release asset) within 24h; doctor refuses next release if prior log not shipped.
4. **No documented evidence retention policy (SOC 2 CC1.4, ISO A.5.33).** Add §12: `.release-audit/` retained ≥7 years, off-laptop replication required.
5. **Tool Trust pin refresh lacks SLA + attestation source (SLSA build.L4 hermeticity).** SR-10 says "explicit PR" but no rule mandates pins originate from upstream-signed checksums. Add R-27: pin updates must cite upstream signed checksum URL in PR body; reviewer asserts checksum chains to known release key.
