# Security Panel — Brutus findings (5)

> **Crash-survival cache.** Promoted to SECURITY-PANEL.md after all 4 agents return.

## #1 CRITICAL — R-21 canary tests the wrong ruleset

**Attacked claim:** P9 + R-21 — "second-channel canary push rejected proves v-tags-signed is active."

**Attack:** The canary push targets `refs/canary/*` (or some non-tag ref). `v-tags-signed` by name + design covers `refs/tags/v*`. The canary's rejection comes from an UNRELATED rule (push-protection, branch-protection, force-push). Attacker silently disables `v-tags-signed`, leaves other rules intact → canary still rejected → spec concludes "ruleset active" → false green. Theater.

**Fix:** R-21 MUST push an unsigned lightweight tag matching `refs/tags/v*` (e.g., `v0.0.0-canary-$(date +%s)`); parse rejection message for the EXACT rule name `v-tags-signed`. Mismatched rule-name in rejection = treat as ruleset-disabled.

## #2 HIGH — R-10 attestation chains to ref name, not GPG-signed tag

**Attacked claim:** "cryptographic proof the wheel was built by our CI from the signed tag."

**Attack:** `gh attestation verify` chains to the REF NAME that triggered the workflow (`refs/tags/v<version>`) + commit SHA at trigger time. It does NOT check the tag is GPG-signed — that's purely a git object property. Anyone with `repo` write (which R-9 mandates) could in principle force-push `v<version>` to an attacker commit, trigger publish.yml, get a perfectly valid attestation bound to that ref. The GPG signature is in the git object; the OIDC claim is just the ref string.

**Fix:** Either constrain to a tag ref AND assert `gh attestation verify --predicate-type ... | jq '.subject.digest.sha1 == $EXPECTED_SHA'` to bind to the SHA captured in P1.5/R-19. Or strike "from the signed tag" from R-10 wording — claim only what's cryptographically true.

## #3 HIGH — §11 Tool Trust pins binary, not runtime (LD_PRELOAD)

**Attacked claim:** "P11 hashes the resolved binary path for each external tool, compares to pins in SR-10."

**Attack:** (a) TOCTOU — pin is computed once at P11; attacker `mv`s binary after the check, before the `gpg --sign` invocation minutes later. (b) `LD_PRELOAD=/tmp/evil.so gpg ...` or `DYLD_INSERT_LIBRARIES` hijacks runtime — binary hash unchanged, runtime owned. (c) P11 doesn't actually refuse binaries under `$HOME` (the documented threat).

**Fix:** (a) re-hash binary immediately before each crypto invocation. (b) `unset LD_PRELOAD LD_LIBRARY_PATH DYLD_INSERT_LIBRARIES DYLD_LIBRARY_PATH DYLD_FORCE_FLAT_NAMESPACE` at script start; assert remain unset before each gpg/gh call. (c) Resolve binaries to absolute paths under `/usr/bin`, `/opt/homebrew/bin` only; refuse `$HOME`-rooted.

## #4 MED — "Offline-capable" claim is misleading

**Attacked claim:** "Keeps script offline-capable + auditable + defends against tag-message injection." (§10 item 1)

**Attack:** Linear paste is generated locally — fine. But every other phase calls `gh api` heavily (P9 canary, R-24 polls, 4.2 pip index, 4.2b attestation verify hits Sigstore TUF/Rekor, 4.7 workflow run). "Offline-capable for Linear paste" is technically true but operationally meaningless. Sets wrong expectation; maintainer on flaky hotspot thinks "Linear is local, no problem" while attestation verify fails and bricks the release.

**Fix:** Strike "offline-capable" from §10 item 1. Replace with "in-process generation (no Linear API coupling) keeps comment auditable + injection-defended; broader release flow requires GitHub + PyPI + Sigstore reachability."

## #5 MED — R-3 watchdog has race window if BOTH watchdog + heartbeat die together

**Attacked claim:** "heartbeat the watchdog; SIGKILL on it = disarmed failsafe = abort" (R-3 / F-4 fix).

**Attack:** Heartbeat detects watchdog death via `kill -0` poll every 5s. Race: attacker (or kernel OOM-killer) SIGKILLs the watchdog at T=0; heartbeat sees at T≤5s. But what if the heartbeat ITSELF is SIGKILLed at T=0 by the same OOM event? Both subprocesses gone. Parent mid-R3..R5. No failsafe fires. The wall-clock watchdog (R-22) was the failsafe; heartbeat was failsafe-for-failsafe; no third level.

**Fix:** Either (a) use OS scheduler as outer watchdog (`systemd-run --user --on-active=120s` / `launchd` LaunchAgent — script can't kill the OS). Or (b) acknowledge limit honestly: add to R-3 "if both watchdog AND heartbeat die simultaneously, recovery is undefined; `release-doctor.sh` MUST be run from a separate shell within 120s — document in runbook."
