Breakdown May 2026
The credential leak hall of shame: $350M, 23,000 orgs, 10 incidents
It's not just billing horror. Leaked credentials bring down enterprises. Here's the taxonomy.

Everyone's seen the "$82K Gemini bill in 48 hours" posts. Those are bad. But they're also recoverable — a provider waiver, a credit card dispute, a support ticket. What follows are the incidents where leaked credentials didn't just cost money. They brought down companies, exposed millions of people, and in some cases, haven't been fully resolved years later.

Category 1: LLM billing abuse (small victims, fast damage)

This is the one developers recognize. Your .env hits GitHub, a scanner finds it within minutes, and spend hammers your key until the bill is too big to ignore. The LLM providers set the table for this by removing hard spend caps and offering generous free tiers that accelerate automated abuse.

Canonical examples: €54K charged in 13 hours (Apr 2026) · $82K Gemini bill in 48 hours (Mar 2026) · $128K GCP bill from one leaked key · $98K Firebase bill in 1 day (May 2025). These aren't outliers — r/googlecloud billing horror is a permanent stickied fixture. OpenAI quietly removed hard spend caps in Dec 2025. Anthropic's "final query" policy allows billing past your prepaid balance.

What makes it survivable: Providers sometimes waive charges. The damage is usually financial, not reputational. You have exactly one lever: revoke the key fast.

Category 2: Infrastructure credentials (medium victims, lasting damage)

These are the incidents where the key wasn't an LLM API key — it was an AWS key, a GitHub OAuth token, a CI secret. The attacker doesn't burn money. They read data.

Capital One (2019) — An SSRF vulnerability in a misconfigured WAF let an attacker query the AWS metadata endpoint and retrieve IAM credentials. Those credentials had over-provisioned S3 access. 106 million customer records exfiltrated. $80M OCC fine + $190M class-action settlement. The root cause wasn't a WAF bug — it was credentials that shouldn't have existed with that scope.

Toyota (2022) — A GitHub token giving access to a customer data system was committed to a public repository and sat there for five years before discovery. 260,000 customers' data was accessible the entire time. Nobody at Toyota noticed. The scanner that found it wasn't Toyota's.

Uber (2022) — An attacker obtained a contractor's Uber credentials, then used MFA fatigue (spamming push notifications until the contractor accepted). From there: full network access, Slack admin, HackerOne bug bounty reports, internal dashboards. All of it. The initial credential wasn't stolen from a codebase — it came from a data broker. But without that credential being over-provisioned, the blast radius would have been a contractor's account, not an entire company.

Category 3: Platform and supply chain (large victims, systemic damage)

Here the credential belongs to a platform that other companies trust. When it leaks, every downstream customer is exposed — and they may not know for months.

Codecov (2021) — Attackers modified Codecov's bash uploader script to exfiltrate environment variables — including CI secrets — from every CI run that used it. The compromise lasted 60 days before discovery. 23,000+ organizations were silently sending their secrets to an attacker-controlled endpoint. Customers didn't know. Codecov didn't know. The attacker did.

CircleCI (2023) — An infostealer on an engineer's laptop captured session tokens with 2FA bypass. Attackers used those tokens to access production systems and exfiltrate customer secrets stored in CircleCI. CircleCI's response: tell every customer to rotate every secret immediately. Approximately one million engineers received that message. The secrets had been encrypted — but CircleCI also stored the encryption keys, so the encryption was theater.

Heroku / Travis CI (2022) — A stolen OAuth user token from Heroku's systems let attackers access private GitHub repositories belonging to GitHub customers using "Login with Heroku." This included npm's private packages. The token was valid and used. Customers were not breached via their own credentials — they were breached because a service they trusted had unrevoked tokens with too much scope.

PyTorch torchtriton (2023) — A malicious package named torchtriton was published to PyPI, shadowing PyTorch's internal package. It executed on install: exfiltrated SSH keys, git config, environment variables, and /etc/passwd to an attacker-controlled domain. 2,717 downloads before removal. Most of those machines are developer workstations — full of credentials, tokens, and keys.

Category 4: Vault compromise (catastrophic, indirect)

The most alarming category: the breach doesn't expose keys directly. It exposes the thing that contains the keys.

LastPass (2022–2023) — Attackers compromised a DevOps engineer's home workstation, which had access to a LastPass shared folder used by the SRE team. That gave them the decryption keys for encrypted LastPass production backups. The backups contained customer vault data. The vaults are still encrypted with customer master passwords — but attackers can crack them offline, forever, at their own pace. Security researchers have linked $35M+ in cryptocurrency losses to decrypted LastPass vaults. The breach compounds over time.

Okta (2023) — Stolen credentials for Okta's support case management system let attackers view HAR files uploaded by customers for debugging. HAR files routinely contain session tokens. 134 Okta customers were affected. Because Okta is an SSO provider, those session tokens could be used against any service the customers used Okta to protect. A $60M securities class-action was filed against Okta in 2024 for alleged concealment of the breach scope.

The pattern

Every single one of these incidents follows the same sequence: credential exists → credential is over-privileged → credential is exposed → scope of access is realized too late.

Vaults help. Rotation policies help. Secret scanning helps. None of them address the underlying problem: a credential that can be reconstructed from a single exfiltrated artifact is a single point of failure for everything it can access. The only architecture that removes the single point of failure is one where no single artifact is sufficient.

That's what Worthless does for API keys. Split-key XOR: neither half is useful alone. No vault to breach because there's no single key to steal.

The incidents above? Most of them aren't solvable by vaults. They're solvable by not having credentials that reconstruct from a single point of access.
