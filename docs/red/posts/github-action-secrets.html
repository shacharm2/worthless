Mar 2025 / CI logs

A GitHub Action printed the secrets.

The build log is supposed to be boring. In this case, it became a place to read secrets after a trusted GitHub Action went bad.

Source: Wiz

what happened

Wiz reported that the compromised `tj-actions/changed-files` GitHub Action exposed CI/CD secrets in workflow logs. That is a bad leak shape: it lands somewhere searchable, copyable, and outside the original trust boundary.

What gets stolen is whatever the workflow had access to: deployment tokens, API keys, package tokens, or cloud credentials. The consequence is usually rotation work, incident response, and fear that someone copied the value before cleanup.

how it got there

The attacker reached a trusted CI dependency. Teams included that action in normal workflows, so the bad behavior ran in places that already had secrets available.

worthless angle

If a locked supported AI-key value appears in logs, copying it elsewhere should not be enough. If the log contains a real provider key, rotate it.
