Why I built Worthless — make API keys worthless post leak

The problem nobody solved

API keys leak. Constantly. Not because developers are careless, but because the entire AI ecosystem runs on plaintext secrets in .env files, and every tool in the chain is designed to move fast and break things.

AI agents run git add . and push your .env to GitHub. CI pipelines print keys in logs. Package managers get compromised and exfiltrate environment variables. And every week, someone discovers their $10k OpenAI bill came from a key they thought was safe.

How many keys are out there right now?

Search GitHub for sk-proj- (OpenAI's key prefix). You'll find tens of thousands of results. Live, working API keys sitting in public repositories. Some have been there for months. Some are being used right now by someone who isn't the owner.

OpenClaw leaks are massive. GitGuardian's annual report says millions of secrets are exposed on GitHub every year. And those are just the ones they can detect.

And it's getting worse. Google's Gemini API keys weren't even considered secrets until Gemini changed the rules. Suddenly, API keys that were safe to commit became attack vectors overnight. Every developer who followed Google's own guidance had keys exposed.

The solutions don't work

Vaults protect keys at rest. Great. But the moment you use the key, it's in memory, in environment variables, in places where it can leak. A vault doesn't help when your CI pipeline logs the key, or when a compromised npm package reads process.env.

Key rotation? You're playing whack-a-mole. Rotate after every leak, hope you catch it in time, pray the attacker didn't rack up a bill in the 47 seconds between leak and rotation.

Then there's the Cloudflare approach: build an entire infrastructure layer with Workers, KV stores, custom auth, rate limiting, and hope your team maintains it. It works, but it's complex enough that most teams won't do it. And the ones that do spend weeks on it.

What if leaked keys just didn't work?

That's the question I couldn't stop thinking about. Not "how do we prevent leaks" (you can't) but "how do we make leaks harmless."

The answer turned out to be simple: split the key. Your .env gets one half. Your server gets the other. Neither half works alone. When you make an API call, the halves combine for a split second, the call goes through, and the key is destroyed.

If someone steals your .env, they get one half of an XOR split. Mathematically useless. They can't brute-force it. They can't derive anything from it. It's just random bytes.

That's Worthless

Two commands. Your key is split. Your .env now contains a share that's worthless to steal. The proxy handles reconstruction, enforces spend caps, and never lets the full key leave server memory.

No vault. No complex infrastructure. No key rotation dance. Just a key that doesn't work when stolen.

Every tool protects your key. None protect you after it leaks

Let's get this out of the way: the tools below are good. Some are great. You should probably use them. This isn't a "we're better" post. It's a "we do a different thing" post.

What everyone does

The entire secrets management industry is built around one idea: keep the key from leaking. Encrypt it at rest. Inject it at runtime. Scan for it in commits. Rotate it when it slips through. And honestly? That works. Most of the time.

Until it doesn't. And it doesn't more often than anyone admits.

1Password / Doppler / Infisical

Solid vaults. You store your secrets in them, they inject into your environment at runtime. 1Password even has CLI integration that's genuinely nice to use. Doppler syncs across environments. Infisical is open source and self-hostable.

The catch: your key still ends up in process.env. A compromised package reads it. A bad console.log prints it. The vault did its job perfectly. The key still leaked.

GitGuardian / GitHub Secret Scanning

These scan your repos for leaked secrets and alert you. GitGuardian catches keys in commits, PRs, even Slack messages. GitHub's built-in scanning works on push and can block the commit entirely.

Genuinely useful. Use them. But they're reactive. They tell you after the key is out there. You still need to rotate, pray nobody used it in the window, and figure out how it slipped through.

Cloudflare's approach

Cloudflare recently showed their API key protection setup. It's impressive: Workers, KV stores, custom auth layers, rate limiting, the works. A proper engineering effort.

But look at it. That's a serious infrastructure project. How many teams are going to build and maintain that? Most of us are shipping features, not building security infrastructure from scratch. If you're Cloudflare, you have the team for it. If you're three people and a Slack channel, you don't.

HashiCorp Vault

The gold standard for secrets management. Dynamic secrets, lease-based access, audit logging, the whole thing. If you're running a 200-person engineering org with a dedicated platform team, Vault is probably the right answer.

If you're a solo dev or a small team building with AI APIs, Vault is like buying a forklift to move a chair.

So where does Worthless fit?

All of the above protect the key before it leaks. Worthless protects you after it leaks.

Your key gets split. Your .env has one half. The proxy has the other. Neither works alone. The full key exists for microseconds during a request, then it's gone.

Use a vault AND Worthless. Use GitGuardian AND Worthless. They're complementary, not competing. The vault prevents leaks. Worthless makes leaks harmless.

Two commands:

pip install worthless
worthless lock

That's the entire setup. No infrastructure to build. No team to maintain it. Your key is split, your .env is safe to leak, and you get back to shipping.

Introducing Worthless — post-leak API key protection

Every developer knows the feeling. You paste an API key into your code "just for testing," it ends up in a git commit, and seconds later it's live on GitHub. Bots scan repos constantly. Your bills spike. You rotate. Repeat next month.

What exists today

Two kinds of tools deal with API key security:

  1. Pre-leak prevention — Vault, dotenv, GitHub secret scanning. These keep keys out of code and logs.
  2. Post-leak response — rotating the key, monitoring for abuse, revoking tokens.

What's missing is the middle: what if the key leaks, but it's already useless?

How Worthless works

Worthless splits your API key into two cryptographic shares. Neither share reveals anything about the original key — you need both to reconstruct it. Reconstruction happens in memory only, for milliseconds, and only when your app actually needs to make an API call.

$ worthless lock
1 key(s) protected.

$ worthless wrap python app.py
# your code is unchanged

$ worthless unlock
1 key(s) restored.

There's also a budget enforcement layer: if your spending limit is hit, the key simply never forms — even if your code tries to use it. A hard stop.

"Every secrets tool protects the key until your app uses it. Worthless protects you after it leaks."

Get started

pip install worthless
worthless lock
worthless wrap python app.py

Why splitting a key makes it useless — the cryptography in plain English

When we say "a leaked share is worthless," we mean it in a precise mathematical sense. This post explains why — without requiring a cryptography background.

The intuition: a lock with two keys

Imagine a padlock that requires two keys simultaneously to open. If someone steals one, they can't open the lock. They have something real — but it's useless without the other half. Secret sharing is the mathematical equivalent.

The XOR method (2-of-2 sharing)

  1. Generate a truly random string A the same length as your secret S
  2. Compute B = S XOR A
  3. Store A locally, B in the proxy

Reconstruction: S = A XOR B. Simple. Brilliant.

Why is A useless alone?

Because A is purely random. An attacker with only A sees that every possible value of S produces a valid B — the share A is statistically independent of S. It could have come from any secret equally well.

Information-theoretic security. This is called one-time pad secret sharing. It cannot be broken even with unlimited computational power — no brute force is possible because every guess is equally valid.

What Worthless does in practice

When you run worthless lock: your key is read, Share A is generated randomly, Share B is derived so A XOR B = key. A goes to local encrypted storage; B lives in proxy memory. Your .env is updated to point at the proxy. The full key exists for milliseconds, then memory is zeroed.

Two independent protections:
1. Cryptographic split — a leaked share reveals nothing.
2. Budget enforcement — even with full key access, spending is capped.
