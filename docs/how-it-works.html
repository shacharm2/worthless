How it works

Your key gets split. Leaked half does nothing.

01

You have an API key in your .env

Right now it's sitting there in plaintext. If anyone gets that file, they have your key. They can use it. You pay.

but why is that bad?

Your .env gets committed to git, printed in CI logs, read by compromised packages, or pushed by an AI agent running git add -A. It happens constantly. Once the key is out there, anyone can use it and you get the bill.

02

Worthless splits it in half

Run worthless lock. Your key gets split into two random-looking pieces. One stays in your .env. The other goes to a local proxy on your machine. Neither piece works on its own.

Someone steals your .env? They get random bytes. Can't use it. Can't brute-force it. Mathematically impossible.

wait, how is it impossible?

It's called XOR secret sharing. Think of it like tearing a photo in half — each piece looks like random noise. You need both halves to see the picture. The math guarantees that one half reveals zero information about the original. Not "hard to guess" — literally zero. Even with every computer on earth running forever.

03

Your app works exactly the same

Run worthless wrap python app.py instead of python app.py. That's the only change. Your code reads environment variables like before. The proxy combines both halves for a split second, makes the API call, then destroys the key.

what happens under the hood?

The proxy sits between your app and the API provider. When your app makes a request, the proxy grabs both halves, XORs them back together, makes the real API call, gets the response, wipes the key from memory, and passes the response to your app. The full key exists for milliseconds. Your app never sees it.

04

Spending cap stops the bleeding

Set a dollar limit. Hit it, the proxy stops reconstructing the key. Hard stop. Your app can't override it. Even if someone somehow gets both halves, the budget gate fires first.

how is this different from OpenAI's spending limit?

OpenAI's limit is checked after the request. Worthless checks before the key even forms. If you're over budget, the key never gets reconstructed. No API call happens. No money spent. The gate is at the infrastructure layer, not the provider's billing page.

05

Don't like it? One command to undo

Run worthless unlock. Original .env restored. Proxy gone. Shares deleted. Like it was never there.

is there really zero lock-in?

Zero. Your original key is stored encrypted locally. worthless unlock decrypts it, writes it back to .env, stops the proxy, and deletes the shares. Your app goes back to reading the real key directly. No artifacts, no config files, no dependencies left behind.
