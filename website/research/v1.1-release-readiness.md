### B3 — Known Security Bugs That Would Embarrass on HN (Critical)

Open P1/Urgent security issues that an auditor would find in hours:

| ID | Issue | Why it matters |
|----|-------|---------------|
| WOR-134 / worthless-3sd | Fernet key persists as plaintext string in proxy memory | RCE = full key compromise. The product CLAIMS memory zeroing. |
| WOR-103 | Auth collapse from alias inference + Shard A fallback | Attacker on localhost can proxy through without auth |
| WOR-104 | OpenAI and Anthropic metering gaps | Budget enforcement is provider-incorrect — spend cap is the headline feature |
| WOR-138 | Key material exposure mitigations (umbrella) | Multiple vectors for key leakage |

These are not theoretical. A security-focused reviewer will find WOR-134 in the first code read. The README says "information-theoretic security" — shipping with plaintext Fernet keys in memory is indefensible.

- **Fix**: At minimum, document known limitations honestly in README. Ideally fix WOR-134 and WOR-104 before launch.
- **Effort**: WOR-134 fix = 4-8 hours. Honest documentation = 1 hour.

## Honest Assessment

The core product works. `worthless lock && worthless wrap python myscript.py` does what it says. The CLI is clean, the help text is good, the security model is real. This is a solid PoC.

But the packaging is not ready for public consumption. The quickstart assumes `git clone`. There is no PyPI package. There are known security bugs in areas the README explicitly claims are handled. There is no changelog, no contribution guide, and zero GitHub discoverability metadata.

The gap between "works on my machine" and "credible open source project" is about 2-3 focused days of work. None of it is technically hard — it is packaging, documentation, and honest communication about known limitations.

The security bugs (B3) are the hardest call. You can either:
- **Fix them first** (adds 1-2 days, launches cleaner)
- **Document them honestly** (adds 1 hour, launches faster, risks credibility if someone finds them before you fix them)

Recommendation: Fix WOR-134 (Fernet key in memory) before launch — it directly contradicts the README's security claims. Document the rest with a "Known Limitations" section and a public timeline for fixes.
